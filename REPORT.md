## 1. Постановка сценария

Клиент отправляет `POST` на оплату. Ответ не доходит из‑за обрыва сети. Клиент повторяет тот же HTTP‑запрос (типичный retry в SDK или двойной клик). Без защиты сервер снова выполняет бизнес‑логику списания: при «unsafe» реализации оплаты возможна вторая запись `paid` в `order_status_history` (двойное списание по смыслу демонстрации ЛР2).

Повтор возможен, потому что на уровне HTTP это новый запрос: сервер не различает «намерение пользователя», если не передан внешний идентификатор операции (идемпотентный ключ) и не сохранён результат первого вызова.

## 2. Реализация таблицы `idempotency_keys`

Итоговый DDL — в файле `backend/migrations/002_idempotency_keys.sql`.

| Поле | Назначение |
|------|------------|
| `idempotency_key` | Значение заголовка `Idempotency-Key` |
| `request_method`, `request_path` | Вместе с ключом задают «логический endpoint» (один ключ на один URL+метод) |
| `request_hash` | SHA‑256 тела запроса; отличает повтор того же намерения от другого payload при том же ключе |
| `status` | `processing` → владелец запроса; `completed` — ответ сохранён |
| `status_code`, `response_body` | Кэш HTTP‑ответа (тело хранится как JSON с полями `body_b64`, `media_type`) |
| `expires_at` | TTL для фоновой очистки старых ключей |

Ограничение `UNIQUE (idempotency_key, request_method, request_path)` не даёт создать две активные записи для одного логического вызова и используется для разруливания гонки «два параллельных первых запроса» (вставка + `IntegrityError` + повторное чтение).

Индексы: по `expires_at` (cleanup), составной lookup по ключу/методу/пути (дублирует уникальность, но явно зафиксирован в миграции).

## 3. Реализация middleware

Файл: `backend/app/middleware/idempotency_middleware.py`.

1. Обрабатываются только `POST` и путь из белого списка (`/api/payments/retry-demo`).
2. Если заголовка `Idempotency-Key` нет — запрос проходит без изменений.
3. Тело читается один раз, запрос пересобирается для downstream (иначе `call_next` не увидит body).
4. Считается `request_hash = SHA256(body)`.
5. В отдельной сессии БД: выборка существующей строки по `(key, method, path)`.
   - Если hash другой → `409 Conflict`.
   - Если статус `completed` → возвращается кэш (заголовок `X-Idempotency-Replayed: true`), бизнес‑хендлер не вызывается.
   - Если `processing` — короткое ожидание и повторная проверка (второй клиент ждёт владельца).
   - Если записи нет — `INSERT` в статусе `processing` (при гонке — `IntegrityError`, затем повтор с шага 5).
6. Владелец вызывает `call_next`, считывает тело ответа, сохраняет `status_code` и кэш в БД, статус `completed`.
7. При необработанном исключении downstream выполняется `DELETE` строки `processing`, чтобы клиент мог повторить попытку с тем же ключом после ошибки.

## 4. Демонстрация без защиты

Тест: `app/tests/test_retry_without_idempotency.py::test_retry_without_idempotency_can_double_pay`.

- Две параллельные попытки `POST /api/payments/retry-demo` с `mode=unsafe` без `Idempotency-Key`.
- На PostgreSQL в истории появляется не менее двух записей со статусом `paid` (инвариант «одна оплата» нарушен на уровне демонстрации ЛР2).
- На SQLite тест пропускается (гонка unsafe надёжно не моделируется из‑за особенностей блокировок SQLite).

Дополнительно: последовательные два вызова без ключа после первой успешной оплаты дают одну запись `paid` в истории (вторая попытка отражается как `success: false` в JSON ответа).

## 5. Демонстрация с `Idempotency-Key`

Тест: `test_retry_with_same_key_returns_cached_response`.

- Первый `POST` с заголовком `Idempotency-Key` оплачивает заказ (`unsafe`).
- Второй идентичный запрос: тот же JSON, тот же ключ → тот же JSON‑ответ, заголовок `X-Idempotency-Replayed: true`, в истории ровно одна запись `paid`.
- В таблице `idempotency_keys` строка в статусе `completed`, в `response_body` есть сохранённый кэш (в т.ч. `body_b64`).

## 6. Негативный сценарий

Тест: `test_same_key_different_payload_returns_conflict`.

Один и тот же `Idempotency-Key`, но во втором запросе другой `order_id` → HTTP 409, повторная оплата не выполняется.

## 7. Сравнение с решением из ЛР2 (`FOR UPDATE`)

Во 2-й лабораторной тесты `test_concurrent_payment_unsafe` / `test_concurrent_payment_safe` демонстрируют параллельную гонку на уровне сервиса оплаты и снятие её через блокировку строки в PostgreSQL. В 4-й сравниваются те же идеи с точки зрения HTTP: повтор запроса после обрыва сети vs защита в БД при повторе без ключа идемпотентности.

| | REPEATABLE READ + `FOR UPDATE` (ЛР2) | Idempotency-Key (ЛР4) |
|---|-------------------------------------------|---------------------------|
| Цель | Корректная сериализация конкурентных транзакций к одной строке заказа в БД | Идентификация логически одного HTTP‑запроса при повторах с клиента |
| Класс угроз | Две параллельные сессии/воркера одновременно меняют одну строку | Последовательный retry того же запроса (обрыв сети, двойной клик с тем же ключом) |
| Сценарий в лабах | `test_concurrent_payment_*`: два параллельных вызова `pay_order_*` | `test_retry_*`, `test_compare_*`: два `POST` на API (с ключом / без, сравнение с `for_update`) |
| Повтор после обрыва | Новый запрос — новая транзакция: обычно получаем бизнес‑ошибку «уже оплачен», а не «тот же ответ» | Тот же ключ и тело → тот же ответ, клиенту не нужно различать «тот же платёж» и «ошибку дубликата» |
| Где гарантия | Уровень СУБД (блокировка строки, изоляция) | Уровень API/HTTP контракта + персистентный кэш ответа |
| UX | Строже с точки зрения клиента (разные коды/тела при повторе) | Предсказуемый повтор успешного ответа |

Они не взаимоисключающие: идемпотентность закрывает сценарий повторной доставки одного запроса; `FOR UPDATE` защищает от параллельных разных запросов к одному заказу (два разных ключа, два таба, два воркера). В продакшене обычно используют оба механизма.

### 7.1. Сопоставление фактических прогонов (ЛР2 vs ЛР4)

Ниже — не теория из кода, а как выглядят логи pytest при типичном прогоне обеих лаб на одной машине (PostgreSQL; ЛР2 — хост и порт `localhost:5433`, ЛР4 — Docker Compose и тесты `retry-demo` / сравнение).

| Аспект | ЛР2 (`lab_02`, `pytest …/test_concurrent_payment_*.py`) | ЛР4 (`lab_04`, `docker compose run --rm backend pytest …`) |
|--------|-------------------------------------------------------------|-------------------------------------------------------------------|
| Что гоняем | Два параллельных вызова сервиса оплаты одного заказа (`unsafe` / `safe`) | Два параллельных `POST /api/payments/retry-demo` без ключа или с одним `Idempotency-Key`, плюс сравнение ветки `for_update` |
| Unsafe / без защиты | В логе: «RACE CONDITION DETECTED», в `order_status_history` две записи `paid` с разными `changed_at` | В логе: предупреждение о двух `paid` в истории, `paid_events=2` — тот же смысл: инвариант «одна оплата» нарушен |
| Safe / FOR UPDATE | «RACE CONDITION PREVENTED», одна `paid`, вторая попытка — `OrderAlreadyPaidError` | В `test_compare_*`: для режима `for_update` две параллельные попытки без ключа — одна успешная оплата в истории, вторая отвергнута на уровне бизнес-логики (в выводе: первая OK, вторая `success=false`, ровно одна `paid`) |
| Новый слой в ЛР4 | В ЛР2 нет HTTP-кэша повтора | С ключом идемпотентности: вторая идентичная пара `ключ+тело` не гоняет оплату повторно — кэш (`X-Idempotency-Replayed`), в истории снова одна `paid`; при другом теле с тем же ключом — 409 |
| Последовательный сценарий | В ЛР2 акцент на параллели; отдельный «два раза подряд» не обязателен | `test_sequential_retry_without_key_*`: два вызова подряд без ключа после первой оплаты — одна запись `paid` (вторая попытка не добавляет вторую оплату) |

Вывод по сопоставлению прогонов. Поведение unsafe в ЛР4 (двойная `paid` в истории при двух параллельных `POST` без ключа) по смыслу совпадает с демонстрацией unsafe в ЛР2 (две записи в истории при двух параллельных платежах). Поведение ветки `for_update` в тесте сравнения ЛР4 совпадает с идеей safe в ЛР2: конкуренция за строку заказа сериализуется, вторая транзакция не оформляет вторую оплату. Отличие ЛР4 в том, что добавлен отдельный контракт HTTP: при повторе того же запроса клиент получает тот же ответ из кэша по ключу, что уже не задача чисто транзакционной модели ЛР2.

## 8. Выводы

1. Повтор запроса после таймаута сети — нормальная ситуация; без идемпотентности это отдельное «намерение оплатить» с точки зрения сервера.
2. Идемпотентный ключ должен быть связан с хэшем тела, иначе клиент может случайно переиспользовать ключ для другой операции.
3. Уникальный индекс + обработка `IntegrityError` — простой способ корректно обработать параллельные «первые» запросы с одним ключом.
4. `FOR UPDATE` не заменяет идемпотентность на границе HTTP и наоборот: у них разные классы угроз.
5. Кэш ответа целесообразно хранить в БД рядом с метаданными ключа и ограничивать по `expires_at`.
